# auto_trading 미진입 사유 분류표 v0.1

작성일: 2026-03-17  
문서 상태: Draft

---

# 1. 문서 목적

본 문서는 데일리 리포트 2-2차 `[놓친 기회]` 섹션 구현을 위해, 당일 전략 후보가 실제 신규 매수 주문으로 이어지지 않은 사유를 어떻게 분류할지 정의한다.

본 문서의 목적은 다음과 같다.

- 미진입 사유를 일관된 코드와 문구로 관리
- 현재 코드에서 바로 집계 가능한 사유와 추가 구현이 필요한 사유를 구분
- 2-2차 구현 범위를 명확히 고정

---

# 2. 전제 조건

미진입 사유는 기본적으로 다음 흐름에서 발생한다.

1. 전략 후보 산출
2. 진입 가능 여부 판단
3. 주문 제출 시도
4. 주문 접수 / 거절 / 미확정

즉, `[놓친 기회]`는 단순히 점수가 높은 종목이 아니라, 다음 조건을 만족하는 종목만 대상으로 본다.

- 당일 후보군에 포함되었음
- 진입 검토 대상이 되었음
- 실제 신규 포지션 진입으로 이어지지 않았음

---

# 3. 현재 코드 기준 사유 분류

## 3.1 바로 집계 가능한 사유

### `max_positions`

의미:

- 최대 보유 종목 수 제한으로 신규 진입이 차단됨

근거 코드:

- `src/auto_trading/risk/engine.py`
- `RiskEngine.can_enter()`

현재 상태:

- 진입 차단 사유 문자열로 바로 반환됨
- 2-2차에서 가장 먼저 포함하기 쉬운 사유

권장 문구:

- `보유 종목 수 한도 도달`

### `failsafe_blocked`

의미:

- Fail-safe 상태로 신규 주문 자체가 차단됨

근거 코드:

- `src/auto_trading/orders/engine.py`
- `src/auto_trading/failsafe/monitor.py`
- `event_type = order_blocked`

현재 상태:

- 주문 제출 단계에서 `order_blocked` 이벤트로 남음
- 현재 리포트에는 별도 집계되지 않음

권장 문구:

- `Fail-safe 차단 상태`

### `already_holding`

의미:

- 동일 종목 active position이 이미 있어 신규 진입을 건너뜀

근거 코드:

- `src/auto_trading/orders/engine.py`
- `event_type = duplicate_position`

현재 상태:

- 실제로는 "중복 포지션 방지" 이벤트로 기록됨
- 리포트용 사유 코드는 `already_holding`으로 매핑하는 것이 이해하기 쉬움

권장 문구:

- `이미 보유 중인 종목`

### `order_rejected`

의미:

- 진입 주문을 시도했지만 브로커가 거절함

근거 코드:

- `src/auto_trading/orders/engine.py`
- `event_type = order_rejected`

현재 상태:

- 이벤트와 실패 메시지가 남음
- 예: 초당 거래건수 초과, 주문 금액 오류 등

권장 문구:

- `주문 거절`

보조 표기:

- 상세 사유 메시지를 함께 보여주는 것이 좋음
- 예: `주문 거절 - 초당 거래건수 초과`

### `order_unknown`

의미:

- 주문 요청 결과가 모호하여 체결 여부를 확정하지 못함

근거 코드:

- `src/auto_trading/orders/engine.py`
- `event_type = order_unknown`
- `event_type = unknown_order_unresolved`

현재 상태:

- 미진입으로 단정하기 전에 "미확정 주문"으로 분리해서 보여주는 것이 안전함

권장 문구:

- `주문 상태 미확정`

---

## 3.2 현재는 바로 집계하기 어려운 사유

### `duplicate_signal`

의미:

- 같은 종목이 반복 후보로 나왔지만 새 진입 대상으로는 의미 없는 경우

현재 상태:

- 별도 이벤트/집계 코드 없음
- 사실상 `already_holding`과 일부 겹침

판단:

- 2-2 1차 범위에서는 제외 권장

### `score_dropped`

의미:

- 후보였지만 이후 점수 하락으로 주문 단계까지 가지 않은 경우

현재 상태:

- 전략 후보 변화 이력과 최종 탈락 시점이 별도로 기록되지 않음

판단:

- 별도 추적 로직이 필요하므로 2-2 1차 범위에서는 제외

### `insufficient_cash`

의미:

- 현금 부족으로 주문 수량 0 또는 주문 실패

현재 상태:

- 현재 RiskEngine이 현금 부족 사유를 독립 코드로 반환하지 않음
- 브로커 거절 메시지로만 일부 드러날 수 있음

판단:

- 추후 `RiskEngine.target_order_size()`와 주문 응답 사유를 함께 정리한 뒤 추가 검토

# 4. 2-2차 1차 구현 범위 추천

2-2차 첫 구현은 아래 5개 사유만 포함하는 것을 권장한다.

- `max_positions`
- `failsafe_blocked`
- `already_holding`
- `order_rejected`
- `order_unknown`

이 범위가 적절한 이유는 다음과 같다.

- 현재 코드/이벤트 구조로 실제 집계 가능
- 운영자가 이해하기 쉬움
- 메시지 길이가 과도하게 길어지지 않음
- 추후 세분화가 쉬움

---

# 5. 권장 리포트 포맷

2-2차에서 `[놓친 기회]` 섹션은 아래 형태를 권장한다.

```text
[놓친 기회]
- 삼성중공업(010140) | 점수=78 | 사유=보유 종목 수 한도 도달
- 한화생명(088350) | 점수=74 | 사유=이미 보유 중인 종목
- 현대지에프홀딩스(005440) | 점수=72 | 사유=주문 거절 - 초당 거래건수 초과
```

표시 규칙:

- 최대 3~5건만 노출
- 점수 높은 순 우선
- 같은 종목 다중 사유는 마지막 결정 사유 1개만 표기
- `order_unknown`은 경고성 사유로 별도 라벨 사용 가능

---

# 6. 구현 메모

2-2차 구현 시 권장 집계 순서는 다음과 같다.

1. 당일 `strategy_snapshots` 또는 후보 리스트에서 대상 종목 수집
2. `RiskEngine.can_enter()` 결과로 `max_positions` 수집
3. `order_blocked`, `duplicate_position`, `order_rejected`, `order_unknown`, `unknown_order_unresolved` 이벤트 수집
4. 종목별 최종 사유 1건으로 정리
5. Telegram 데일리 리포트와 수동 데일리 리포트에 동일 포맷 반영

우선순위 규칙 권장안:

1. `order_rejected`
2. `order_unknown`
3. `failsafe_blocked`
4. `already_holding`
5. `max_positions`

주문 단계 사유를 상위 우선순위로 두는 이유는, 실제 운영 복기 관점에서 가장 직접적인 실패 원인이기 때문이다.
